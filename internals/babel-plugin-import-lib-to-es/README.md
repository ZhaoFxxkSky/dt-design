# babel-plugin-import-lib-to-es

[English](./README.md) · [简体中文](./README-zh_CN.md)

A Babel plugin that rewrites `<pkg>/lib` subpath imports to `<pkg>/es` at compile time — so that the ESM build always consumes the ES module graph of upstream packages.

```diff
- import { Button } from 'foo-lib/lib/button';
+ import { Button } from 'foo-lib/es/button';
```

## Features

- **Compile-time rewrite** — `<pkg>/lib` → `<pkg>/es`, no runtime cost.
- **Any package, zero config** — rewrites every `/lib` subpath out of the box; new dependencies require no configuration.
- **Scope-aware** — correctly handles `@scope/name/lib/...` (captures the full scoped package name before matching `include` / `exclude`).
- **Pure & deterministic** — never touches the file system. Output is independent of `cwd` and `node_modules` layout, so builds are reproducible across environments.
- **Plays well with `babel-plugin-import`** — runs on `Program.exit`, so imports generated by `babel-plugin-import` in the same pass are also rewritten.
- **Comprehensive syntax coverage** — `import`, `export ... from`, `export * from`, bare `require()`, and dynamic `import()` are all handled.

## When to Use

Use this plugin when producing an **ESM** build of a library whose source references CommonJS subpaths (`/lib`) of upstream packages. Such paths typically come from:

1. **Source code** that directly imports internals, e.g. `import { Config } from 'foo-lib/lib/config'`.
2. **`babel-plugin-import` output** — when it expands `import { Button } from 'foo-lib'` into `foo-lib/lib/button`.

Shipping `/lib` paths in an ESM build forces bundlers to load the CommonJS module graph alongside the ES one, which causes:

- **Duplicated module instances** — the same module loaded from `/lib` (CJS) and `/es` (ESM) are two different instances; for libraries relying on module-level singletons (e.g. React Context), this breaks the singleton guarantee.
- **Broken Tree Shaking** — CJS modules are not statically analyzable; the entire CJS graph is bundled.
- **Inconsistent module format** — an ESM build should form a single, consistent ES module graph.

> **Do not** register this plugin for the CommonJS (`lib`) build. The CJS build must keep `/lib` paths intact.

## Usage

Register the plugin **only** for the ESM build environment. The CJS build must keep `/lib` paths intact.

### Via `babel.config.js`

```js
module.exports = (api) => {
  const isESM = api.env('esm');

  return {
    plugins: [
      isESM && [
        require.resolve('./internals/babel-plugin-import-lib-to-es'),
        // optional: { exclude: ['lodash'] }
      ],
    ].filter(Boolean),
  };
};
```

### Via `.babelrc`

```json
{
  "env": {
    "esm": {
      "plugins": [["./internals/babel-plugin-import-lib-to-es", { "exclude": ["lodash"] }]]
    }
  }
}
```

## Options

| Option | Type | Default | Description |
| --- | --- | --- | --- |
| [`exclude`](#exclude) | `Array<string \| RegExp>` | `[]` | Package names whose `/lib` paths should **not** be rewritten. |
| [`include`](#include) | `Array<string \| RegExp>` | — | Strict mode: if provided, **only** these package names are rewritten. Takes precedence over `exclude`. |

By default the plugin rewrites **any** package's `/lib` path. If a rewritten package has no `/es` layout, resolution fails loudly at build time — add it to `exclude`.

### `exclude`

Package names to leave untouched. Accepts strings (exact match) or `RegExp` (tested against the captured package name).

```js
// lodash has no /es layout — exclude it
[
  './internals/babel-plugin-import-lib-to-es',
  { exclude: ['lodash'] },
]

// exclude by regex
[
  './internals/babel-plugin-import-lib-to-es',
  { exclude: [/^lodash/] },
]
```

### `include`

Switches to strict mode: only the listed package names are rewritten. `include` takes precedence over `exclude`.

```js
// Only rewrite packages under a specific scope
['./internals/babel-plugin-import-lib-to-es', { include: [/^@my-scope\//, /^rc-/] }];
```

## What Gets Rewritten

### Path patterns

| Pattern                | Rewrite? | Example                                        |
| ---------------------- | -------- | ---------------------------------------------- |
| `<pkg>/lib`            | ✅       | `foo-lib/lib` → `foo-lib/es`                   |
| `<pkg>/lib/<sub>`      | ✅       | `foo-lib/lib/button` → `foo-lib/es/button`     |
| `@scope/pkg/lib/...`   | ✅       | `@scope/pkg/lib/hooks` → `@scope/pkg/es/hooks` |
| `<pkg>` (package root) | ❌       | `foo-lib` — untouched                          |
| `<pkg>/<other>`        | ❌       | `lodash/merge` — untouched                     |
| Relative paths         | ❌       | `./lib/utils` — untouched                      |

### Supported syntax

The rewrite applies to the `source` of the following AST nodes:

| AST Node                            | Example                                       |
| ----------------------------------- | --------------------------------------------- |
| `ImportDeclaration`                 | `import X from 'foo-lib/lib/button'`          |
| `ExportNamedDeclaration`            | `export { default } from 'foo-lib/lib/hooks'` |
| `ExportAllDeclaration`              | `export * from 'foo-lib/lib/locale'`          |
| `CallExpression` (bare `require`)   | `const X = require('foo-lib/lib/button')`     |
| `CallExpression` (dynamic `import`) | `import('foo-lib/lib/button')`                |

## How It Works

The plugin registers a `Program.exit` visitor that traverses the AST and rewrites the `source` string of import / export / require / dynamic-import nodes.

Running on `Program.exit` (rather than `enter`) is deliberate: it ensures imports **generated** by other plugins earlier in the same pass — most notably `babel-plugin-import`, which expands `import { Button } from 'foo-lib'` into `foo-lib/lib/button` on its own `Program.exit` — are also rewritten.

```
Source                        babel-plugin-import            This plugin
──────                        ────────────────────            ──────────────
import { Button }                                            import { Button }
  from 'foo-lib';      →      from 'foo-lib/lib/button';  →   from 'foo-lib/es/button';

import { Config }                                            import { Config }
  from 'foo-lib/lib/config';                                from 'foo-lib/es/config';
                                                             (Program.exit)
```

## Design Principles

- **Pure & deterministic** — the plugin never touches the file system. Output does not depend on `cwd` or `node_modules` layout, making builds reproducible across environments.
- **Scope-aware** — correctly handles `@scope/name/lib/...` (captures the full scoped package name before matching `include` / `exclude`).
- **Zero runtime cost** — all rewrites happen at compile time; the published artifacts contain only `/es` paths.
- **Fail fast** — if a rewritten package has no `/es` layout, resolution fails loudly at build time rather than silently producing broken artifacts.

## Recipes

### Default: Rewrite Any Package

The default behavior rewrites any `/lib` path with zero configuration — ideal when all upstream dependencies ship an `/es` layout:

```js
['./internals/babel-plugin-import-lib-to-es'];
```

### Monorepo with Custom Package Scope

If your monorepo publishes under a custom scope and only those packages should be rewritten:

```js
['./internals/babel-plugin-import-lib-to-es', { include: [/^@my-scope\//] }];
```

### Co-operating with `babel-plugin-import`

Register `babel-plugin-import` **before** this plugin so that the `/lib` paths it generates are rewritten in the same pass:

```js
plugins: [
  ['import', { libraryName: 'foo-lib', style: false }],
  ['./internals/babel-plugin-import-lib-to-es'],
];
```

## FAQ

<details>
<summary><strong>Should I enable this plugin for my CJS (CommonJS) build?</strong></summary>

No. The CJS build must keep `/lib` paths intact. Register the plugin only for the ESM build environment.

</details>

<details>
<summary><strong>What happens if a rewritten package has no <code>/es</code> layout?</strong></summary>

Module resolution fails loudly at build time with an error like `Cannot find module 'lodash/es/merge'`. Add the package to `exclude`:

```js
{
  exclude: ['lodash'];
}
```

</details>

<details>
<summary><strong>Does the order of plugins matter?</strong></summary>

Yes. If you use `babel-plugin-import`, register it **before** this plugin. `babel-plugin-import` generates `/lib` paths on `Program.exit`; this plugin also runs on `Program.exit` but later in the visitor queue, so the generated paths are picked up.

</details>

<details>
<summary><strong>Does it rewrite <code>require()</code> and dynamic <code>import()</code>?</strong></summary>

Yes. Bare `require('foo-lib/lib/button')` and dynamic `import('foo-lib/lib/button')` are both rewritten. Member calls such as `foo.require(...)` are left untouched.

</details>

<details>
<summary><strong>Will it touch relative paths like <code>./lib/utils</code>?</strong></summary>

No. Only the `/lib` segment directly after a package name is rewritten. Relative paths and package-root imports (`import 'foo-lib'`) are left untouched.

</details>
